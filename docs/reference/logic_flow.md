# ⚙️ 로직 흐름 및 비즈니스 규칙 (v8.1+)

본 문서는 `뇌사료` 프로젝트 내부에서 작동하는 특수 로직들과 데이터 처리 규칙을 상세히 설명합니다.

---

## 🔒 1. 보안 및 암호화 관리 로직

### 1.1 암호화 상태 전이 (Security Lifecycle)
- **보안 마스킹**: 암호화된 메모(`is_encrypted=1`)는 본문 조회 API 요청 시 백엔드에서 내용이 아스테리크(`***`)로 마스킹되어 내려옵니다.
- **해독 상태 유지**: 사용자가 비밀번호를 입력하여 해독에 성공하면, `unlockedMemos` 상태 배열에 해당 ID를 등록하여 세션 동안은 다시 묻지 않고 평문을 보여줍니다.
- **수정 시 검증**: 해독된 상태에서 메모를 수정(`PUT`)하더라도, 보안상의 이유로 Body에 비밀번호를 반드시 포함해야 서버에서 재암호화 후 저장됩니다.

### 1.2 첨부파일 보안 (Forbidden 해결)
- **세션 검증**: 이미지 및 파일 다운로드 요청 시 `auth.login_required` 데코레이터를 통해 유효한 세션 쿠키를 가진 사용자만 접근을 허용합니다.
- **동적 경로 보정**: 본문 내 이미지 경로가 상대 경로(`uploads/file.png`)인 경우, 렌더링 시 `fixImagePaths` 유틸리티가 `/api/download/file.png`로 자동 변환하여 403 에러를 방지합니다.

### 1.3 첨부파일 생명주기 및 고아 파일 자동 정리 (Attachment Lifecycle & Cleanup)
- **개별 삭제 및 세션 관리**: 작성 도중 업로드한 파일들은 프론트엔드에서 `sessionFiles`로 추적되며, 작성 취소 시 즉각적으로 서버 삭제 요청을 보내 잉여 파일을 방지합니다.
- **서버 자동 청소 (Garbage Collection)**: 브라우저 강제 종료 등으로 인해 미처 지워지지 못한 고아 파일(작성 중단 파일)은 서버 백엔드 기동(`init_db`) 시점에 DB의 `memo_id IS NULL` 레코드를 스캔하여 물리적 파일과 함께 완전히 삭제됩니다.

## 🌌 2. 지식 관계 시각화 (Visual Graph Logic)

### 2.1 VisualLinker: 드래그 앤 와이어링
- **실시간 좌표 추적**: `Alt + 드래그` 시 시작점(Source)과 마우스 커서 사이의 거리를 계산하여 SVG `line` 또는 `path` 요소를 실시간으로 업데이트합니다.
- **데이터 바인딩**: 드롭 성공 시 대상 메모의 ID를 본문에 `[[#ID]]` 형식으로 즉시 삽입하고 서버에 저장합니다.

### 2.2 RelationManager: 관계 포커스 모드
- **하이라이트 알고리즘**: 특정 메모 선택 시, 해당 메모를 Source 또는 Target으로 가지는 모든 `memo_links` 데이터를 필터링합니다.
- **시각적 격리**: 연결되지 않은 다른 메모 카드와 화살표는 투명도(`opacity`)를 낮춰 사용자가 현재 맥락에만 집중할 수 있도록 처리합니다.

## 🔄 3. 데이터 흐름 및 필터 동기화 정책

### 3.1 AppService 중심의 단방향 흐름
1.  **UI Event**: 사용자가 필터를 변경하거나 데이터를 요청합니다.
2.  **Service Action**: `AppService` 파사드가 `FilterService.setFilter()`를 호출하여 `AppState`를 변경하고 `API`와 통신합니다.
3.  **Broadcasting**: 데이터 로드가 완료되면 `AppService`는 등록된 모든 매니저(Weekly, Sidebar, Layout 등)의 렌더링 콜백을 순차적으로 실행합니다.

### 3.2 지능형 필터 초기화
- **영역 우선순위**: 사이드바의 주요 그룹(별표, 완료 등)을 클릭하면 세부 필터(날짜, 카테고리)는 초기화되지만, 검색어(`query`)는 유지하여 사용자가 현재 보던 맥락에서 검색을 이어갈 수 있도록 합니다.

## 🛰️ 4. 외부 동기화 및 UUID 식별 로직 (External Sync)

### 4.1 UUID 기반 식별 (v8.2+)
- **영구 주소 확보**: 내부 ID(Integer)는 DB 재구축 시 변경될 수 있지만, UUID는 생성 시 부여되어 삭제 전까지 변하지 않는 영구적인 식별자입니다.
- **외부 연동**: Obsidian 등 외부 앱은 항상 UUID를 통해 메모를 참조하며, 동기화 시 충돌 방지의 기준점이 됩니다.

### 4.2 마크다운 변환 엔진
- **Frontmatter**: 외부 API 호출 시, 메모의 메타데이터(제목, 태그, 날짜 등)를 YAML Frontmatter로 변환하여 본문 상단에 결합합니다.
- **역변환**: 외부에서 수정된 마크다운을 전송하면, 서버는 Frontmatter를 파싱하여 메타데이터를 업데이트하고 본문만 추출하여 저장합니다.

## 📅 5. 주간 뷰 활동량 연동 로직
- **데이터 공유**: `WeeklyManager`는 렌더링 시 전역에 노출된 `window.HeatmapManager.data`를 직접 참조합니다.
- **레벨링**: 해당 날짜의 메모 수(`count`)에 따라 0(없음)에서 4(매우 많음)까지의 클래스(`lvl-0` ~ `lvl-4`)를 부여하여 글래스모피즘 도트 색상을 결정합니다.

## 🚀 6. 배포 및 복구 자동화 (Ops)

### 5.1 원격 배포 (deploy.py)
- **Surgical Update**: `.git` 정보와 `data/`, `static/uploads/` 등 사용자 자산은 보존하고 소스 코드 파일(`app/`, `static/`, `templates/` 등)만 수술적으로 교체합니다.
- **환경 변수 보존**: `.env` 파일은 절대 삭제하거나 덮어쓰지 않아 배포 후에도 DB 연동 및 비밀번호 설정이 유지됩니다.

### 5.2 재난 복구 (DR)
- **백업 3요소**: `memos.db`, `uploads/` 폴더, `.env` 파일만 있으면 언제 어디서든 서버를 동일하게 재구성할 수 있도록 설계되었습니다.
