## 🔍 상세 분석 및 해결 방안

### 1. D3.js 리소스 누수 (Memory / CPU Leak)
> [!WARNING]
> 사용자가 그래프 뷰어(Visualizer)를 여러 번 열고 닫을 때, 이전 렌더링에 사용된 물리 연산(Simulation)이 백그라운드에서 종료되지 않고 계속 실행됩니다.

*   **영향도 (Medium)**: 브라우저 탭의 CPU 점유율이 지속적으로 상승하며, 랩탑의 경우 배터리 소모를 유발합니다. 앱을 오래 켜둘수록 눈에 띄게 느려질 수 있습니다.
*   **수정 난이도 (Low)**: 기존 시뮬레이션 객체를 참조(Ref)로 저장해두고, 새로운 렌더링이 시작되거나 모달이 닫힐 때 `simulation.stop()`을 호출하는 코드 2~3줄만 추가하면 해결됩니다.

### 2. 전역 알림(Toast Notification) 시스템 부재
> [!NOTE]
> API 통신 에러나 권한 오류 시 브라우저 기본 `alert()` 창을 사용하여 작업 흐름을 강제로 끊고 있습니다.

*   **영향도 (High)**: 투박한 UX를 제공하며, `alert`가 떠 있는 동안 자바스크립트 메인 스레드가 멈추기 때문에 비동기 작업 로직이 꼬일 수 있습니다.
*   **수정 난이도 (Medium)**: 화면 모서리에 부드럽게 나타나고 사라지는 `ToastManager.js`를 신규 생성한 뒤, 기존 `alert(err.message)` 코드를 일괄 검색하여 `ToastManager.error()`로 치환해야 합니다.

### 3. 상태(State)와 렌더링(UI)의 강한 결합
> [!IMPORTANT]
> 상태(`AppService.state`)가 변경되었을 때 화면이 자동으로 갱신되지 않아, 개발자가 수동으로 `LayoutManager.renderMemos()`를 호출하여 동기화하고 있습니다.

*   **영향도 (High)**: 새로운 기능을 추가할 때마다 "데이터는 업데이트했는데 왜 화면은 안 바뀌지?" 하는 동기화 버그가 빈번하게 발생할 수 있는 근본 원인입니다.
*   **수정 난이도 (High)**: 프론트엔드 전반의 아키텍처를 손봐야 합니다. 가벼운 이벤트 버스(Event Bus)나 발행/구독(Pub-Sub) 패턴을 도입하여, 상태 변경 시 이벤트를 발생시키고 UI가 알아서 렌더링되도록 리팩토링해야 합니다.

### 4. 컴포넌트 소멸자(Lifecycle Management) 부재
> [!TIP]
> 모달, 에디터 등 대부분의 컴포넌트가 생성(`init`) 로직만 있고, DOM에서 제거되거나 닫힐 때 이벤트 리스너를 정리하는 소멸(`destroy` / `cleanup`) 로직이 없습니다.

*   **영향도 (Medium)**: 장시간 앱을 리프레시 없이 사용할 경우 이벤트 리스너가 중복 등록되거나, 참조를 잃은 DOM 노드(고아 노드)가 메모리에 쌓여 느려집니다.
*   **수정 난이도 (Medium)**: 당장 큰 오류를 내지는 않지만, 표준화된 컴포넌트 생명주기 규칙을 세우고 주요 모달들에 일일이 `cleanup()` 메서드를 구현해야 하므로 시간이 소요됩니다.
