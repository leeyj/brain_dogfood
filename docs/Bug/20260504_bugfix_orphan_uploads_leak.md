# 버그 수정 내역 (2026-05-04)

## 개요
- **이슈:** 메모 작성 또는 수정 중 파일을 업로드한 뒤, 저장을 하지 않고 취소하거나 창을 닫을 때 업로드된 파일이 서버 디스크 및 DB에 "고아(Orphan)" 상태로 무한히 누적되는 메모리 릭 발생.
- **원인 분석:**
  1. **프론트엔드 누락:** 에디터를 완전히 닫는 `Discard(작성 취소)` 버튼 외에, 단순히 '새 메모 작성'을 다시 누르거나 기존 에디터를 '접기(Fold)'만 했을 경우, `ComposerManager`가 에디터 상태를 초기화(`clear()`)하면서 세션에 쌓여있던 임시 파일 추적 내역(`sessionFiles`)을 서버 삭제 요청 없이 그대로 증발시켜버림.
  2. **백엔드 누락:** 브라우저를 강제 종료하거나 새로고침했을 때는 프론트엔드가 삭제 API를 호출할 수단 자체가 없어, 연결된 메모가 없는(NULL) 찌꺼기 레코드가 방치됨.

## 영향도
- 작성 취소 시 삭제되어야 할 더미 파일들이 `static/uploads`를 가득 채워 서버의 디스크 용량과 DB 자원을 낭비함.

## 수정 사항
1. **프론트엔드 (ComposerManager.js 개선):**
   - 메모 작성을 새로 시작(`openEmpty`)하거나 기존 메모를 편집기로 불러올 때(`openForEdit`), **기존에 진행하다가 중단된 임시 세션이 있다면 먼저 서버에 삭제 요청을 보내 깨끗하게 청소**(`EditorManager.cleanupSessionFiles()`)하도록 로직 추가.
   - 기존 메모 삭제(`deleteBtn`) 시 해당 세션에 추가되었던 파일들도 함께 처리되도록 누락된 구문 보완.

2. **백엔드 (database.py 개선):**
   - 브라우저 종료나 새로고침 등 예외 상황으로 인해 방치된 파일들을 구제하기 위해 **"서버 재시작 자동 청소 로직"** 도입.
   - 서버가 구동되면서 `init_db()`를 호출할 때, `attachments` 테이블에서 `memo_id IS NULL`인 레코드를 모조리 색인하여 **실제 물리적 파일을 디스크에서 삭제하고 DB 레코드도 삭제**함.

## 해결 완료
- 이제 프론트엔드에서의 작성 취소/이탈 대응은 물론, 뜻밖의 브라우저 튕김이 발생해도 서버를 한 번 껐다 켜면 찌꺼기 파일이 자동으로 완전 소멸하여 C 드라이브 용량 부족 사태를 근본적으로 예방함.
