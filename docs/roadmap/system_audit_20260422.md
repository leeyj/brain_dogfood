# 시스템 구조 진단 및 리팩토링 로드맵 (v8.1+)

## 1. 개요
현재 시스템은 v8.1 리팩토링을 통해 대대적인 모듈화가 진행되었으나, 일부 전역 상태 의존성과 렌더링 효율성 측면에서 개선의 여지가 남아 있습니다. 신규 기능의 안정적인 확장을 위해 다음 사항들을 검토합니다.

## 2. 리팩토링 항목 및 진행 상황

| 항목 | 설명 | 우선순위 | 상태 | 기대 효과 |
| :--- | :--- | :---: | :---: | :--- |
| **전역 캐시 정리** | `window.allMemosCache` 등 window 객체 의존성 제거 및 AppService 통합 | **상** | ✅ **완료** | 데이터 흐름의 명확성 확보 및 디버깅 용이성 |

## 3. 상세 진단 내용

### 3.1 프론트엔드 (JavaScript)
- **전역 오염**: `window.UI`, `window.memoEventHandlers`, `window.allMemosCache` 등이 여전히 브릿지 역할로 사용되고 있어 모듈 캡슐화를 저해함.
- **렌더링 부하**: `AppService.refreshData()` 호출 시 Masonry 레이아웃 전체를 다시 계산하므로, 메모 수가 많아질수록 성능 저하 우려.
- **결합도**: `AppService`가 `CalendarManager`, `HeatmapManager` 등 하위 매니저의 내부 메서드를 직접 호출하고 있음.

### 3.2 백엔드 (Python/Flask)
- **데이터 전송 효율**: `Visualizer`에서 그래프를 그릴 때 메모의 전체 본문까지 포함된 데이터를 가져오고 있어 페이로드 낭비 발생.
- **보안 로직**: 암호화된 메모 삭제 시 해독을 강제하는 로직에 대한 사용자 정책 검토 필요.

## 4. 향후 실행 계획
1. **Short-term**: 전역 캐시 정리 및 시각화 API 최적화 진행.
2. **Mid-term**: 부분 업데이트 로직 구현으로 UX 극대화.
3. **Long-term**: 이벤트 기반 아키텍처로 완전한 디커플링 완성.
